# Logo Hump Count

Date: 2026-03-25
Status: Canonical branding rationale

The camel logo uses **6 humps**.

Why `6`:

1. `AMS / BIOS orchestration`
   This is the command core: sessions, agent pool behavior, spawning, GSD flow, and orchestration discipline.
2. `Subscription runtime matrix`
   The project is defined by real operator surfaces for Codex, Claude, and Kimi across desktop, CLI, and VS Code.
3. `Domain state contract`
   Tasks, roadmaps, contexts, Merkle state, and repository/schema alignment hold the durable project memory together.
4. `Tool and router bus`
   Tool execution, routing, middleware, and runtime handoff form the internal movement system of the project.
5. `Operator surfaces`
   CLI, dashboard, and WebSocket surfaces turn the core into a controllable product instead of a hidden library.
6. `Observability and release control`
   Security, alerts, monitoring, workflows, Pages, and release checks keep the repository honest and shippable.

What does **not** count as a hump:

- individual providers by themselves
- single controllers or repositories
- isolated utilities
- one-off reports or planning artifacts
- visual design assets

The hump count is meant to stay stable unless the project’s real architectural load-bearing abstractions change.
